# 基于LangChain的RAG系统优化

## 一、RAG商业化痛点分析

### 1. 索引构建阶段

| 痛点                    | 表现                         |
| :---------------------- | :--------------------------- |
| **内容缺失**            | 知识库原本无答案             |
| **文档加载准确性/效率** | PDF等复杂格式提取困难        |
| **文档切分粒度**        | 过大丢失语义，过小破坏上下文 |

### 2. 查询回答阶段

| 痛点                  | 表现                        |
| :-------------------- | :-------------------------- |
| **错过Top Ranked**    | 正确答案不在召回的前K个块中 |
| **Not in Context**    | 召回的上下文与问题无关      |
| **格式错误**          | 输出格式不符合预期          |
| **答案不完整**        | 只回答部分问题              |
| **未提取到答案**      | 上下文有答案但LLM未提取     |
| **答案过于具体/笼统** | 特异性失衡                  |

## 二、Advanced RAG 三层优化体系

text

```
Naive RAG → Advanced RAG → Modular RAG
                ↓
         Pre-Retrieval + Retrieval + Post-Retrieval
```

### （一）Pre-Retrieval 预检索优化

#### 1. 优化索引结构

| 索引类型           | 核心思路                                              | 适用场景               |
| :----------------- | :---------------------------------------------------- | :--------------------- |
| **摘要索引**       | LLM为每个块生成摘要并向量化，检索摘要后回溯原文档     | 半结构化数据、新闻资讯 |
| **父子索引**       | 小块索引精准匹配，召回后替换为大父块保证上下文完整    | 法律条款检索           |
| **假设性问题索引** | 为每个块生成3个假设问题并索引，用问题向量匹配用户问题 | 药品咨询、复杂查询     |
| **元数据索引**     | 利用标签（如时间、作者）先过滤再向量检索              | 电商推荐、多源文档     |

#### 2. 查询优化策略

| 策略                      | 说明                               |
| :------------------------ | :--------------------------------- |
| **Enrich完善问题**        | LLM主动引导补全用户模糊意图        |
| **Multi-Query多路召回**   | 生成N个相关查询并行检索，汇总结果  |
| **Decomposition问题分解** | 复杂问题拆分为子问题串行或并行求解 |

### （二）Retrieval 检索优化

- **混合检索**：结合向量检索（语义）与关键词检索（精确），适应异构数据与复杂查询
- **嵌入模型微调**：针对专业领域定制嵌入模型

### （三）Post-Retrieval 后检索优化

| 技术                   | 作用                                               |
| :--------------------- | :------------------------------------------------- |
| **RAG-Fusion**         | 多查询结果通过RRF算法重排序，取Top K送入LLM        |
| **重排序 (Reranking)** | 用更精准的模型调整召回文档顺序                     |
| **上下文压缩**         | 提取召回文档中与查询最相关的片段，减少无关信息干扰 |
