### 🔍 两次运行上下文差异的根本原因

对比你贴出的两次输出：

| 运行       | 上下文包含的内容                                             |
| :--------- | :----------------------------------------------------------- |
| **第一次** | 仅 `二、入职` 下的单个段落（约 150 字）                      |
| **第二次** | `一、聘用` 的完整内容 + `二、入职` 的两个子段落（含“不符合录用条件”的 8 条细则，约 600 字） |

**这说明**：两次检索返回的文档集合**完全不同**。第一次只命中了“入职”块，第二次却命中了包含“聘用”和“不符合录用条件”的多个块。

------

### ❓ 为什么 `similarity` 检索仍会返回不同的文档集合？

#### 1. 向量相似度分数极度接近，导致排序漂移

在你的文档库中，可能存在多个与“入职注意事项”语义相关的块：

- 块 A：“入职人员提交资料：身份证、银行卡……”
- 块 B：“凡有以下情形者，均被视为不符合录用条件……”
- 块 C：“聘用程序……”

这些块与查询 `“入职有什么注意事项？”` 的相似度分数可能都在 `0.85~0.87` 之间，差异极小。由于：

- **浮点数精度**：余弦相似度计算涉及高维向量点积，不同硬件/运行次序可能导致小数点后第 5~6 位的差异。
- **HNSW 图搜索的近似性**：Chroma 默认使用 HNSW 索引，它在检索时会进行**近似最近邻搜索**，牺牲微小精度换取速度。每次检索遍历的图路径可能略有不同，导致相似度分数接近的文档排序发生变化。

#### 2. 你设置的 `k=4` 可能恰好处在“临界区”

假设相关文档的相似度排序为：

text

```
1. 块A (入职流程)        score=0.872
2. 块B (提交资料)        score=0.869
3. 块C (不符合录用条件)  score=0.865
4. 块D (聘用程序)        score=0.862
5. 块E (岗前培训)        score=0.860
```

### ✅ 如何稳定检索结果？

| 措施                          | 具体操作                                                   | 效果                           |
| :---------------------------- | :--------------------------------------------------------- | :----------------------------- |
| **1. 增大检索数量并固定排序** | 先取 `k=10`，然后按内容或元数据排序，再截取前 N 个送给 LLM | 消除排序漂移对最终上下文的影响 |
| **2. 强制精确检索**           | 在 Chroma 中增大 `hnsw:search_ef` 参数，关闭近似搜索       | 检索结果高度确定，但速度稍慢   |
| **3. 对文档块进行语义去重**   | 在 `format_docs` 中过滤掉内容高度重复的块                  | 避免冗余块挤占有效上下文       |

#### 稳定化代码修改示例

python

```
# 1. 创建向量库时，强制精确检索
vectorstore = Chroma.from_documents(
    texts, embeddings,
    persist_directory="./chroma_db",
    collection_metadata={"hnsw:space": "cosine", "hnsw:search_ef": 200}  # 值越大越精确
)

# 2. 检索器先多取一些文档，再排序截断
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 10}  # 先取 10 个候选
)

# 3. 格式化时排序并截断（确保确定性）
def format_docs(docs):
    # 按文档内容排序（或按 metadata 中的 ID/来源）
    sorted_docs = sorted(docs, key=lambda d: d.page_content)
    # 取前 5 个作为最终上下文（可调整）
    selected_docs = sorted_docs[:5]
    return "\n\n".join(doc.page_content for doc in selected_docs)
```

### 📝 为什么 LLM 回答仍会因上下文不同而产生差异（即使 temperature=0）？

`temperature=0` 只是让 LLM 在**给定相同输入**时输出确定性的结果。但如果**输入本身（上下文）发生了变化**，LLM 自然会生成不同的回答。因此，**稳定上下文是稳定回答的前提**。

------

### 💡 补充建议：检查向量库是否被重复追加

如果你多次运行脚本且未清空 `persist_directory`，`Chroma.from_documents` 会**追加**新文档而非覆盖，导致向量库中存在大量重复块，加剧相似度打平问题。

**解决方案**：在创建向量库前，先删除旧目录：

python

```
import shutil
import os

persist_dir = "./chroma_db"
if os.path.exists(persist_dir):
    shutil.rmtree(persist_dir)
```

### 💎 总结

| 现象                   | 根本原因                                          | 解决方案                                      |
| :--------------------- | :------------------------------------------------ | :-------------------------------------------- |
| 两次运行上下文内容不同 | 向量相似度接近 + HNSW 近似检索导致 Top-K 排序漂移 | 增大 `hnsw:search_ef`；先取较多候选再排序截断 |
| 回答随之不同           | 上下文变化导致 LLM 基于不同信息作答               | 稳定上下文即可稳定回答                        |

通过以上调整，你的 RAG 系统将获得**完全可复现**的检索结果和回答内容。

### ⚙️ 在 Chroma 中的配置与调优

理解原理后，我们可以通过在 Chroma 中配置 `ef_search` 来解决问题。**注意**，Chroma 1.0及以上版本建议使用 `configuration` 参数。

- **创建集合时配置（推荐）**

  python

  ```
  collection = client.create_collection(
      name="my_collection",
      configuration={
          "hnsw": {
              "space": "cosine",
              "ef_construction": 200,  # 构建时探索广度，影响索引质量
              "ef_search": 200,        # 👈 关键参数，搜索时探索广度
              "max_neighbors": 32,     # 每个节点的最大连接数 (M)
          }
      }
  )
  ```

  **配置项说明**：

  - `ef_search`: 查询时的动态候选列表大小，影响召回率和速度。默认值为100。
  - `ef_construction`: 构建索引时的动态候选列表大小，影响索引构建的质量和速度。默认值为100。
  - `max_neighbors`: 每个节点的最大连接数（M）。值越大，图的连通性越好，但内存消耗也越高。默认值为16。
  - `space`: 距离度量方式，可选 `l2`, `cosine`, `ip` (内积)。

- **调优建议**：

  - 如果你的数据量不大，且对稳定性要求极高，可以尝试将 `ef_search` 设置为一个远大于默认值的数（如 `500` 或 `1000`）。
  - 这种“暴力”方式能让搜索范围覆盖几乎整个图，结果将趋于**完全确定**，但其代价是牺牲查询性能，不适用于大数据量或高并发场景。

如果你想了解在不同场景下如何平衡稳定性和性能，或是对HNSW的其他参数感兴趣，随时可以继续提问。
