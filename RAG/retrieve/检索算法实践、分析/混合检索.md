## 混合检索

将 BM25（关键词检索）与向量检索（语义检索）结合使用，通常称为**混合检索（Hybrid Search）**。核心思路是**融合两种检索方式的优势，取长补短**：

- **BM25**：擅长精确匹配（如专有名词、产品型号、用户输入的确切关键词），不依赖训练数据，计算快，但无法理解同义词或上下文。
- **向量检索**：擅长语义理解（如“苹果”与“iPhone”的关联），能处理近义词和模糊查询，但可能漏掉罕见的精确词或受训练数据偏差影响。

混合检索的目标是让最终结果既“准”（关键词匹配）又“全”（语义相关）。以下是具体实现思路：

------

## 1. 基本流程：并行检索 + 结果融合

最典型的做法是**双路召回**：

1. **并行执行两个检索器**：
   - 将同一个查询分别传给 BM25 检索引擎（如 Elasticsearch、rank_bm25）和向量检索引擎（如 Faiss、Milvus）。
   - 各自返回 top‑K 个文档及其相似度得分（BM25 得分和向量余弦/内积得分）。
2. **融合两个得分列表**：将两路的结果合并成一个最终排序列表。这一步是关键，因为两个得分处于不同量纲和分布（BM25 得分通常是非负且无上界，向量相似度一般在 [0,1] 或 [-1,1] 区间）。

------

## 2. 常用的融合方法

### 方法一：倒数排名融合（RRF, Reciprocal Rank Fusion）

- **原理**：忽略原始得分，只利用文档在两路结果中的**排名**。
- **公式**：对每个文档，计算 `RRF_score = Σ 1/(k + rank_i)`，其中 `rank_i` 是该文档在第 i 个检索器返回列表中的排名（从1开始），`k` 是常数（通常取 60）。
- **优点**：不需要对得分进行归一化，对异常值不敏感，简单高效。
- **适用**：通用场景，尤其当两路检索质量都较高时。

### 方法二：加权得分求和（Weighted Score Summation）

- **步骤**：
  1. 分别对 BM25 得分和向量相似度得分进行**归一化**（例如 Min‑Max 归一化或 Z‑Score 归一化），使两者落在相近区间（如 [0,1]）。
  2. 给两种得分分配权重 `α` 和 `β`（`α + β = 1`），计算最终得分：`final_score = α * norm(bm25_score) + β * norm(vector_score)`。
  3. 按最终得分重排文档。
- **优点**：可以灵活调整权重（如对精确匹配要求高的场景加大 BM25 权重）。
- **缺点**：归一化方法选择和权重调优需要实验验证。

### 方法三：重排序（Re‑ranking）

- **流程**：
  1. 先用一种检索器快速召回一个较大的候选集（例如 BM25 召回 top‑200）。
  2. 再用另一种检索器对候选集进行精细打分（例如用向量模型计算查询与每个候选文档的相似度），然后按新得分排序。
- **常见变体**：
  - **BM25 召回 → 向量重排**：适合 BM25 已经能覆盖大部分相关文档，但需要语义精排的场景。
  - **向量召回 → BM25 重排**：适合语义召回覆盖面广，但需要精确词匹配来纠正错位的场景。
- **优点**：计算量相对小（第二路只需要处理候选集），且可以结合不同模型的能力。

------

## 3. 实际工程中的实现框架

以 **RAG（检索增强生成）** 系统为例，典型代码结构（伪代码）：

python

```
# 1. 双路检索
bm25_results = bm25_retriever.search(query, top_k=50)   # 返回 [(doc_id, bm25_score), ...]
vector_results = vector_retriever.search(query, top_k=50) # 返回 [(doc_id, vector_score), ...]

# 2. 融合（以 RRF 为例）
k = 60
fusion_scores = {}
for rank, (doc_id, _) in enumerate(bm25_results, start=1):
    fusion_scores[doc_id] = fusion_scores.get(doc_id, 0) + 1/(k + rank)
for rank, (doc_id, _) in enumerate(vector_results, start=1):
    fusion_scores[doc_id] = fusion_scores.get(doc_id, 0) + 1/(k + rank)

# 3. 排序输出
sorted_docs = sorted(fusion_scores.items(), key=lambda x: x[1], reverse=True)
final_docs = [doc_id for doc_id, score in sorted_docs[:10]]
```

如果使用加权求和，则需要先归一化得分：

python

```
def normalize(scores_list):
    min_s = min(scores_list)
    max_s = max(scores_list)
    return [(s - min_s) / (max_s - min_s + 1e-8) for s in scores_list]

norm_bm25 = normalize([s for _, s in bm25_results])
norm_vec = normalize([s for _, s in vector_results])
alpha = 0.5
final_scores = {doc_id: alpha * nb + (1-alpha) * nv 
                for (doc_id, _), nb, nv in zip(bm25_results, norm_bm25, norm_vec)}
```

## 4. 进阶思路：学习型融合

如果拥有标注数据（查询‑相关文档对），可以训练一个**排序模型**（如 LambdaRank、LightGBM）来学习融合权重。特征包括：

- BM25 得分、向量相似度得分
- 两个检索器各自的排名
- 查询和文档的长度、词重叠度等

这种方法效果最好，但需要额外的标注数据和训练流程。

------

## 5. 注意事项

- **索引分开**：BM25 使用倒排索引，向量检索使用向量索引（如 HNSW）。两者独立维护。
- **延迟和成本**：双路检索会加倍计算开销，需根据业务 QPS 合理设置 top‑K（召回数量不宜过大）。可以使用缓存或异步优化。
- **参数调优**：RRF 的 `k` 值、加权求和的 `α` 以及各检索器的 top‑K 都需要在验证集上调优。

------

## 总结思路

> **并行执行 BM25 和向量检索 → 融合两者的结果（常用 RRF 或加权得分）→ 输出最终排序文档。**

这种混合检索已经成为现代搜索引擎（如 Elasticsearch 的 RRF 查询）、RAG 系统（如 LlamaIndex、LangChain 中的 `HybridSearchRetriever`）的标准实践。它能同时响应精确关键词和语义意图，显著提升检索的鲁棒性和准确率。
